# phd-thesis-paper-template

Self-contained PhD thesis chapter templates for empirical CS/SE that follow a basic conference paper structure. The goal is to let candidates write the thesis first, then turn each core chapter into a paper with minimal adaptation (roughly one paper per chapter), while leaving room for thesis-specific material.

## Purpose and scope
This repository provides markdown files with guidelines and templates for:
- Building a PhD thesis around conference-style core chapters.
- Ensuring each core chapter covers the minimal elements of a solid empirical paper.
- Making it easy to convert chapters into conference submissions with limited restructuring.

Although the templates originate from a Computer Science / Software Engineering thesis, other fields can adapt the structure to their own conventions.

## Recommended thesis structure
A typical thesis layout supported by these templates is:
- Chapter 1 – Introduction
- Chapters 2–4 – Core chapters, each mapping to a paper-style unit
- Chapter 5 – Conclusions and Outlook
- Optional – Appendices with implementation and technical details

Each core chapter is designed as a self-contained unit, close to a standard empirical paper.

## Core chapter template
Each core chapter follows a basic conference-style structure:
- Introduction
- Background
- Related Work
- Methodology
- Results
- Discussion
- Summary

This layout is meant to:
- Match common empirical CS/SE paper formats.
- Provide clear separation between context, method, findings, and interpretation.
- Make the transition from chapter to paper mostly a matter of trimming, tightening, and adapting to venue constraints.

## How to use these templates
A suggested workflow for PhD candidates:
- Write the thesis chapters first, following the core chapter and introduction/conclusion templates.
- Once a chapter is stable, extract a paper version by trimming non-essential thesis framing, tightening text, and aligning with a specific venue’s requirements.
- Iterate between thesis and paper versions as needed, bringing improvements back into the thesis while avoiding endless rework on unpublished material.

## Contributions and adaptations
Suggestions, issues, and pull requests are welcome. Examples of useful contributions:

- Discipline-specific adaptations outside CS/SE.
